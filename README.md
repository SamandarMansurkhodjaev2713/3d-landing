# ПОЛЮС // Архив миссии

Кинематографичный 3D-лендинг про автономный лунный ровер «П-6», работающий
в районах постоянной тени у кратера Шеклтон (южный полюс Луны). Сюжет и техника
вымышлены; сайт — авторский портфолио-проект.

Технически это WebGL-сцена на чистом Three.js (без React/GSAP/игровых движков),
собранная как fullscreen-canvas с прокруткой: камера летит между шестью ракурсами
по секциям, ровер разбирается и собирается на середине пути, поверхность —
настоящий фотограмметрический реголит, а вступление — короткая срежиссированная
сцена «включение борта → спуск камеры → восход над кромкой кратера».

**🌐 Живая версия:** https://samandarmansurkhodjaev2713.github.io/3d-landing/
(деплоится автоматически из ветки `main` через GitHub Actions — см. [Деплой](#деплой)).

## Стек и архитектура

- **[Vite 5](https://vitejs.dev/)** — сборка и dev-сервер, без плагинов и фреймворков поверх.
- **[Three.js 0.171](https://threejs.org/)** — вся 3D-сцена на чистом API, без R3F/движков.
- **Vanilla JS/CSS** — без React, без GSAP, без CSS-фреймворков. Анимации — CSS-переходы
  и прямые вызовы Three.js API в едином `requestAnimationFrame`-цикле.
- Один общий рендер-цикл ([src/three/animationLoop.js](src/three/animationLoop.js)),
  который останавливается на скрытой вкладке (`visibilitychange`).

Код разбит по ответственности, а не по фичам: `three/` не знает про DOM,
`ui/` не знает про WebGL, `config/` — только данные (никакой логики).

## Быстрый старт

Требуется Node.js 18+.

```bash
npm install
npm run dev
```

Откройте адрес, который выведет Vite (обычно `http://localhost:5173`).

## Структура проекта

```
src/
  config/     данные без логики: пути к ассетам, контент секций, тема,
              ракурсы камеры по секциям, дефолты сцены (свет/экспозиция)
  three/      вся WebGL-часть: рендерер, сцена, камера, свет, небо, грунт,
              загрузка модели/HDRI, камера-риг, композитор постпроцессинга,
              разборка ровера, солнечный диск, пыль, лидар-пинг
  ui/         DOM-слой: навигация, мобильное меню, прогресс-бар, HUD,
              появления секций, счётчики, курсор, хотспоты, панели настройки
  styles/     модульный CSS (по одному файлу на область: layout/sections/
              hud/panels/animations/cursor/hotspots/responsive/…)
  main.js     точка входа: собирает всё выше в один сценарий загрузки и цикл

public/
  assets/
    models/   GLB-модель ровера (DRACO + WebP, см. «Ассеты и лицензии»)
    hdr/      HDRI-окружение (только для отражений/освещения материалов)
    textures/ PBR-текстуры грунта и камней
  draco/      декодер DRACO для GLTFLoader

docs/
  ARCHITECTURE.md   подробный технический разбор: интро, риг камеры,
                     шейдеры грунта, адаптивное качество и т.д.
```

## Как устроена сцена

Коротко — для быстрого понимания; детали и обоснования решений в
[docs/ARCHITECTURE.md](docs/ARCHITECTURE.md).

**Загрузка и вступление.** Пока грузятся модель и HDRI, сцена уже видна сквозь
полупрозрачный лоадер («Cold Boot»): камера дрейфует на широком ракурсе, борт
«не проснулся» (свет и солнечный диск выключены). После загрузки — «Arc Descent»:
камера идёт по дуге к герою, сначала глядя на восход солнца над зубчатой кромкой
кратера, затем поздним рефреймом находит ровер — единственный насыщенный цвет
в кадре загорается ровно с восходом. Хэндофф в риг скролла бесшовный: интро
финиширует точно в позе первого ракурса.

**Камера по скролу.** [cameraRig.js](src/three/cameraRig.js) интерполирует между
шестью ракурсами ([cameraKeyframes.js](src/config/cameraKeyframes.js)) по позиции
скролла, с орбитальным поворотом (yaw) и кадрово-независимым сглаживанием.
Десктоп и мобайл (портрет) используют разные наборы ракурсов — на мобильном ровер
работает как фон под текстом, а не как главный объект кадра.

**Разборка ровера.** Модель (423 меша в глубокой иерархии) «сплющивается» в
плоский список одним проходом ([explode.js](src/three/explode.js)), после чего
каждую деталь можно независимо сдвигать наружу — раскрытие/сборка синхронизированы
со скроллом к секции «Системы».

**Грунт.** Не процедурная заглушка: реальные фотограмметрические PBR-карты
лунного реголита (Poly Haven `moon_meteor_01`) на CPU-деформированной плоскости
с несколькими кратерами, плюс шейдерный грейзинг-блеск по терминатору и
макро-вариация, убирающая видимую сетку тайлинга — всё через `onBeforeCompile`,
без дополнительных проходов рендера.

**Атмосфера.** Небесный купол и звёзды — кастомные `ShaderMaterial` (мерцание,
Млечный Путь, туманности, Земля вдалеке), видимый диск солнца на векторе
ключевого света, редкие пылинки реголита, ловящие низкий свет, реальные тени от
DirectionalLight (десктоп), сдержанный bloom + кино-грейд через `EffectComposer`
(тоже десктоп-only).

**Адаптивность.** Один и тот же код ветвится по `matchMedia('(max-width:768px)')`
на старте: мобайл получает свои ракурсы, меньше сегментов геометрии, меньше
частиц, без теней/bloom/хотспотов. В рантайме — мониторинг FPS: при устойчивой
просадке сначала выключается bloom, затем снижается `pixelRatio`.

## Продакшн-режим и dev-инструменты

В коде есть две панели для тонкой настройки прямо в браузере — «Сцена»
(экспозиция, свет, HDRI) и «Камера» (все параметры текущего ракурса, с кнопкой
«Скопировать значения» для вставки обратно в `cameraKeyframes.js`).

Эти панели **включены только в `npm run dev`** и отсутствуют в продакшн-сборке
(`npm run build`) — в портфолио-версии сайта их не видно. При необходимости
их можно принудительно показать и на собранном сайте через `#dev` в URL,
а в dev-режиме, наоборот, скрыть через `#clean`.

## Производительность

- Модель сжата DRACO (геометрия) + WebP (текстуры): **~57 МБ → ~5 МБ**
  без потери деталей и без изменения структуры мешей.
- `pixelRatio` ограничен (1.1 на мобильном, 1.35 на десктопе) вместо полного DPR.
- Тени, bloom, цветокоррекция и хотспоты инспекции — только на десктопе.
- `logarithmicDepthBuffer` включён точечно ради устранения z-fighting на плотной
  модели при разборке — при добавлении новых постпроцесс-эффектов, работающих
  с depth-буфером (SSAO, DoF), это стоит учитывать.
- Рендер-цикл останавливается на скрытой вкладке.
- Рантайм-адаптив качества по FPS (см. выше).

## Деплой

Проект деплоится на **GitHub Pages автоматически** через GitHub Actions
([.github/workflows/deploy.yml](.github/workflows/deploy.yml)): на каждый пуш в
`main` workflow ставит зависимости, собирает `dist/` и публикует его. Ручной
запуск тоже доступен — вкладка **Actions → Deploy to GitHub Pages → Run workflow**.

**Разовая настройка (один раз для нового репозитория):**
в GitHub → **Settings → Pages → Build and deployment → Source** выбрать
**«GitHub Actions»**. После этого пуши в `main` публикуются сами; статус видно
во вкладке Actions, итоговая ссылка — в логе job `deploy`.

**Про подпапку.** Сайт-проект на GitHub Pages живёт по адресу
`https://<user>.github.io/<repo>/`, то есть в подпапке. Поэтому в
[vite.config.js](vite.config.js) продакшн-`base` = `/3d-landing/`, а все пути к
ассетам в коде строятся через `import.meta.env.BASE_URL` — благодаря этому одна
и та же сборка корректно грузит модель, текстуры и DRACO-декодер и локально
(`/`), и на Pages (`/3d-landing/`). При переименовании репозитория поправьте
`base` (и ссылки на живую версию/OG-изображение).

**Другие хостинги.** Если деплоить на **Cloudflare Pages / Vercel / Netlify**
(корень домена), поставьте `base` обратно в `/` — остальное (build command
`npm run build`, output `dist`) определяется автоматически.
